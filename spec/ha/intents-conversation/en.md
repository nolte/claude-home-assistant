# HA Integration: Intents and Conversation

Status: draft

## Context

An **intent** is the description of a user's intention — a high-level action generated by user actions, such as "asking Amazon Echo to turn on a light." Intents are **fired** by components that receive them from external sources/services (Conversation, Alexa, API.ai, Snips) and can be **handled** by **any** component. That is exactly what makes it easy for a Custom Integration to integrate with all voice assistants at once: it registers an `IntentHandler`, and as soon as a matching intent is fired — regardless of the source — its devices respond through Assist.

Intents are implemented through the `homeassistant.helpers.intent.Intent` class; an intent carries `hass`, `platform`, `intent_type`, `slots`, and optionally `text_input` and `language`. An integration registers a handler via `intent.async_register(hass, handler)`, declares `intent_type` there, optionally a `slot_schema`, and implements `async_handle(intent_obj)`, which returns an `IntentResponse`. Intents can be fired from your own code via `intent.async_handle(hass, platform, intent_type, slots)`; the response is an `IntentResponse` with `speech`, `reprompt`, and `card`.

HA ships a set of **built-in intents** (`HassTurnOn`, `HassTurnOff`, `HassGetState`, …) that an integration can implement so its devices respond to Assist sentences. Through the conversation layer (`/api/conversation/process`), an input sentence is processed into text recognition, intent firing, and a `conversation response`. An integration can additionally provide its **own conversation agent** as a `ConversationEntity`.

This spec addresses the **developer-facing intent/conversation API** — handlers, slots, built-in intent implementation, conversation entity, and sentences. It does **not** address end-user Assist configuration (exposing entities in the UI, voice pipeline setup). Delimitation: intent vs. service is captured in `ha/services`; localization of intent sentences (`intents/<lang>.yaml`) belongs to `ha/translations`; that a conversation entity is a regular entity follows from `ha/entity-architecture`.

## Goals

- Make `IntentHandler` registration the sole entry point for your own high-level actions — one source per `intent_type`
- Implement built-in intents (`HassTurnOn`/`HassTurnOff`/`HassGetState`) so the integration's devices respond to standard Assist sentences, rather than offering only proprietary intents
- Enforce slot validation through a declared `slot_schema`, rather than ad-hoc checking in handler code
- Phrase intent responses consistently through `IntentResponse`/`async_set_speech`, so Assist receives a speakable reply
- Provide custom conversation agents as a `ConversationEntity` with `supported_languages`, rather than re-implementing the deprecated `async_process` pattern
- Separate intent sentences and their localization from handler logic, so language and code stay independently maintainable

## Non-Goals

- End-user Assist configuration (exposing entities in the UI, voice pipeline, STT/TTS setup) — user documentation, not developer API
- Translation of intent sentences (`intents/<lang>.yaml`, response texts) — owned by `ha/translations`
- Service definition and the `services.yaml` schema — separate `ha/services` spec; the intent-vs-service delimitation is captured there and below
- Entity lifecycle and registration of the conversation entity in detail — owned by `ha/entity-architecture`
- External intent sources (Alexa, API.ai, Snips bridge setup) — the spec addresses only the handler side, not source configuration

## Requirements

### Registering intent handlers

- **MUST** derive every custom intent handler from `homeassistant.helpers.intent.IntentHandler` and register it via `intent.async_register(hass, handler)`
- **MUST** set `intent_type` on the handler class as a string naming the intent to handle (for example `intent_type = "CountInvocationIntent"`)
- **MUST** implement `async_handle(self, intent_obj: intent.Intent) -> intent.IntentResponse` and return an `IntentResponse` — never `None` or a raw string
- **SHOULD** register exactly one handler per `intent_type` — multiple handlers for the same type make resolution ambiguous
- **MUST NOT** register intent handlers in platform modules (`sensor.py`, `light.py`); registration belongs in `async_setup` / `async_setup_entry`

### Slots & slot schema

- **MUST** read slot values only through `intent_obj.slots` — the dictionary is keyed by slot name, and the actual value lives under `slots["<name>"]["value"]`
- **SHOULD** declare a `slot_schema` validation schema on the handler class (for example `slot_schema = {"item": cv.string}`) when the intent expects slots, rather than using the input unchecked
- **MUST** pass slots in the documented form when firing an intent from your own code — `slots = {"<name>": {"value": <value>}}` to `intent.async_handle(hass, platform, intent_type, slots)`
- **MAY** evaluate `intent_obj.text_input` and `intent_obj.language` when the handler needs the raw text or input language — both are optional and may be absent

### Implementing built-in intents

- **SHOULD** implement the supported built-in intents that fit the integration's domain (for example `HassTurnOn`, `HassTurnOff`, `HassGetState`), so the devices respond to standard Assist sentences
- **MUST** treat the slots for `HassTurnOn`/`HassTurnOff` as **optional** and support the documented slot combinations (for example `name only`, `area only`, `area and domain`, `device class and domain`)
- **MUST NOT** re-implement **deprecated** intents (`HassOpenCover`, `HassCloseCover`, `HassToggle`, `HassHumidifierSetpoint`, `HassHumidifierMode`, `HassShoppingListLastItems`) — use the respective recommended replacement intents instead (for example `HassTurnOn` instead of `HassOpenCover`)
- **SHOULD** treat the built-in intent catalog as generated from the Intents repository and not maintain a local copy of the intent list that can drift

### Intent response & speech

- **MUST** create the response through `intent_obj.create_response()` and set the spoken reply via `response.async_set_speech("<text>")`
- **MUST** deliver speech responses only in the allowed types — `plain` (the default) or `ssml`; other speech types are not allowed
- **MAY** keep the session open through the `reprompt` response when a user reply is required — in that case `speech` is usually a follow-up question
- **MUST** respect the correct `response_type` of the conversation response — `action_done` (an action was performed, with `targets`/`success`/`failed`), `query_answer` (answer to a question), or `error`
- **MUST** set an appropriate `data.code` in the error case — one of `no_intent_match`, `no_valid_targets`, `failed_to_handle`, `unknown` — and surface the error message through `speech`

### Conversation agent (`ConversationEntity`)

- **MUST** derive a custom conversation agent from `homeassistant.components.conversation.ConversationEntity` — not directly from `AbstractConversationAgent` when the entity variant is available
- **MUST** declare `supported_languages` on the conversation entity (`list[str]` or `"*"` for all languages) — the property is required
- **MUST** process incoming messages in `_async_handle_message(self, user_input, chat_log) -> ConversationResult` and return the result as a `ConversationResult` with `response` and `conversation_id`
- **SHOULD** implement `_async_handle_message` instead of the deprecated `async_process` — the new hook automatically includes the `chat_log` and is backwards compatible
- **SHOULD** set the `CONTROL` feature (`ConversationEntityFeature.CONTROL`) only when the agent is actually able to control Home Assistant
- **MAY** implement `async_prepare(self, language)` to pre-load resources (a language model or similar) as soon as a request is announced — the method is optional
- **MUST NOT** perform I/O (network requests or similar) in property getters — properties return only information from memory

### Sentences & localization

- **SHOULD** maintain intent sentences in `intents/<lang>.yaml`, separating them from handler logic, so language and code stay independently maintainable
- **MUST** respect the input language when processing a conversation input sentence — `language` is optional and otherwise falls back to the configured HA language
- **MAY** pre-load a language's sentences through the WebSocket API (`conversation/prepare`) when first-sentence latency should be reduced
- **MUST NOT** hard-wire localized sentences or response texts into handler code — localization belongs in the language artefacts (`ha/translations`)

## Acceptance Criteria

- [ ] Every intent handler derives from `IntentHandler`, is registered via `intent.async_register(hass, handler)`, and sets `intent_type`
- [ ] `async_handle(intent_obj)` returns an `IntentResponse` created through `create_response()` / `async_set_speech()`
- [ ] Slot values are read through `intent_obj.slots["<name>"]["value"]`; a `slot_schema` is declared for expected slots
- [ ] Domain-appropriate built-in intents (`HassTurnOn`/`HassTurnOff`/`HassGetState`) are implemented; `HassTurnOn`/`HassTurnOff` treat slots as optional
- [ ] No deprecated intents (`HassToggle`, `HassOpenCover`, …) are re-implemented
- [ ] Speech is delivered only as `plain` or `ssml`; errors carry a valid `data.code`
- [ ] The custom conversation agent derives from `ConversationEntity`, declares `supported_languages`, and implements `_async_handle_message(...) -> ConversationResult`
- [ ] `ConversationEntityFeature.CONTROL` is set only when the agent can actually control HA
- [ ] Intent sentences live in `intents/<lang>.yaml`; no localized texts are wired into handler code

## Open Questions

- **`AbstractConversationAgent` vs. `ConversationEntity`**: The docs show both paths. Should the spec strictly mandate `ConversationEntity`, or does `AbstractConversationAgent` remain a permitted alternative for agents without an entity representation?
- **Custom vs. built-in intents**: At what point SHOULD an integration define its own `intent_type` instead of extending a built-in intent? A threshold (for example "only when no built-in covers the action") would make skill output deterministic.
- **`continue_conversation` semantics**: When SHOULD a handler keep the session open (`continue_conversation: true` / `reprompt`)? Currently only described as a possibility — a concrete guideline is missing.
- **Conversation-id tracking**: Should the spec require that every agent support multi-turn through `conversation_id`, or does it stay optional (return `None` when unsupported)?
- **Chat-log usage**: What obligations apply to reading/writing the `chat_log` (tool calls, history)? The typed Python interface is linked, but the spec does not abstract it here yet.
