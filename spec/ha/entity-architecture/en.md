# HA Integration: Entity Architecture

Status: draft

## Context

An entity in a Custom Integration is identified and named through three cooperating mechanisms: a **`unique_id`** (stable identification for the entity registry, never UI-visible), a **`translation_key`-based display name** (translatable display name, working together with `_attr_has_entity_name`), and the **`entity_id` automatically generated by HA** (technical ID in `domain.<slug>` form, derived from device name and entity name at runtime). Manually setting `self.entity_id` is an anti-pattern because it bypasses HA's own slug logic and locks in a language-/installation-dependent ID.

In parallel, HA established the **`EntityDescription` pattern**: instead of writing one entity class per datapoint, the integration declares a tuple list of `<Platform>EntityDescription` instances, and a generic entity class iterates them. That makes the integration declarative rather than imperative: a new entity variant is a tuple-list entry, not a new class.

`nolte/kamerplanter-ha` validates the triple ŌĆö `unique_id` format, `translation_key`-based names, `EntityDescription` pattern ŌĆö consistently across five platforms (sensor, binary_sensor, button, calendar, todo). This spec lifts the convention into a generic obligation. The device hierarchy itself (DeviceInfo, `via_device`) lives in the parallel `ha/device-registry` spec.

Quality scale markers:
- **Bronze**: stable, non-random `unique_id` per entity.
- **Silver**: `_attr_has_entity_name = True` plus `translation_key`-based names instead of hard-coded strings; `EntityDescription` pattern instead of one class per datapoint.

## Goals

- Mandate a `unique_id` per entity that survives re-discovery, server restart, and HA restart
- Make `_attr_has_entity_name = True` the default ŌĆö HA then composes the display name from device name + entity name, with the entity name coming from `translation_key`
- Forbid manual `self.entity_id` assignment ŌĆö HA generates the `entity_id` from the (English) display name automatically
- Establish the `EntityDescription` pattern as the standard form for platform setup, so adding a new entity variant is a data row instead of a new class
- Surface mandatory entity-category markings (`CONFIG`, `DIAGNOSTIC`), so the HA UI sorts entities correctly
- Have generated code start Bronze/Silver conformant straight from skill output

## Non-Goals

- DeviceInfo construction and `via_device` hierarchy ŌĆö separate `ha/device-registry` spec
- `RestoreEntity` pattern (state persistence across restart) ŌĆö separate follow-up spec when concretely needed
- Custom state-class definition (`SensorStateClass.MEASUREMENT` vs. `TOTAL_INCREASING` vs. ŌĆ”) ŌĆö belongs in platform-specific follow-up specs (`ha/sensor-platform`, ŌĆ”); this spec only covers the cross-platform entity architecture
- Lovelace-specific display logic (card layout, custom cards) ŌĆö separate `ha/lovelace-card-patterns` spec
- HA translation format itself (`strings.json` shape, sync to `translations/<lang>.json`) ŌĆö separate `ha/translations` spec

## Requirements

### Base entity class

- **MUST** define a cross-platform base entity class in `entity.py` that subclasses `homeassistant.helpers.update_coordinator.CoordinatorEntity` (or the timestamp variant when used)
- **MUST** set `_attr_has_entity_name = True` as a class attribute on the base class ŌĆö every derived entity inherits it
- **MUST** offer a typed constructor on the base class that accepts at least `coordinator`, `entry_id`, and a `device_info: DeviceInfo`, and assigns `self._attr_device_info = device_info` (see `ha/device-registry`)
- **SHOULD** store `entry_id` as an instance attribute on the base class (`self._entry_id = entry_id`) ŌĆö needed for `unique_id` construction and diagnostics
- **MUST NOT** put platform-specific logic in the base class ŌĆö the base class stays agnostic to sensor / binary sensor / button / ŌĆ”

### `_attr_has_entity_name` and `translation_key`

- **MUST** set `_attr_has_entity_name = True` on every entity (typically inherited from the base class, see above)
- **MUST** set a `translation_key` on every entity ŌĆö either as `_attr_translation_key = "<key>"` or via the `EntityDescription.translation_key` property
- **MUST** ensure the `translation_key` has an entry in `strings.json` under `entity.<platform>.<key>.name` (see `ha/translations`)
- **MUST NOT** set `_attr_name` as a hard-coded string ŌĆö hard-coded names defeat translation and produce the same text in every UI language
- **MAY** set `_attr_name = None` on the **main device entity**, so the entity carries only the device name (typical for the core entity that represents a device)

### `unique_id`

- **MUST** set a `unique_id` on every entity ŌĆö either as `_attr_unique_id = "..."` in the constructor or via the `EntityDescription.key` composition
- **MUST** construct the `unique_id` so it carries entry scope, resource type, resource slug, and a suffix:
  - Format: `{entry_id}_{resource_type}_{resource_slug}_{suffix}`
  - Examples: `{entry_id}_server_refresh_all`, `{entry_id}_<resource>_<slug>_<descriptor>`
- **MUST** keep the `unique_id` stable across the lifetime of an entry ŌĆö an entity called `_phase` today and `_growth_phase` tomorrow is a different entity from the registry's perspective and causes data loss
- **MUST NOT** use a random `uuid` as `unique_id` ŌĆö random generation breaks across restarts
- **MUST NOT** derive the `unique_id` from the `entity_id` ŌĆö `entity_id` is language- and installation-dependent

### Forbidden `entity_id` assignment

- **MUST NOT** set `self.entity_id = "..."` manually ŌĆö HA generates the `entity_id` from the device name and the entity name automatically on registry entry
- **SHOULD** make sure the display name in `strings.json` (English as source of truth) is chosen so the generated `entity_id` is readable ŌĆö the `entity_id` is based on the **system-language display name** at first registration; an English source language produces stable, language-independent IDs (see `ha/translations` for the language convention)
- **MAY** set `_attr_suggested_object_id` in special cases when HA's generated slug is objectively bad; that is a suggestion HA can override on conflict

### `EntityDescription` pattern

- **MUST** use `EntityDescription` tuple lists for every platform with more than two datapoints, instead of one class per datapoint
- **MUST** define the tuple list as a top-level constant: `<DOMAIN>_<PLATFORM>_<ROLE>_DESCRIPTIONS: tuple[<Platform>EntityDescription, ...] = (...)`
- **MUST** set at least `key` and `translation_key` in every `EntityDescription` ŌĆö additional fields (`device_class`, `state_class`, `native_unit_of_measurement`, `entity_category`, `icon` as fallback) per platform
- **MUST** annotate the generic entity class with `entity_description: <Platform>EntityDescription` and assign `self.entity_description = description` in the constructor
- **SHOULD** implement datapoint extraction (`_handle_coordinator_update` or the `native_value` property) generically via `entity_description.key` resolution rather than per-key `if/elif` chains
- **MAY** keep a single dedicated class instead of `EntityDescription` for tightly coupled special entities (for example a single refresh button)

### Entity categories

- **SHOULD** mark entities that **change configuration** (refresh button, mode selector, threshold number) as `EntityCategory.CONFIG` ŌĆö the HA UI then lists them under "Configuration" rather than "Sensors"
- **SHOULD** mark entities that surface **read-only diagnostic information** (sensor-offline status, last-update timestamp, firmware version) as `EntityCategory.DIAGNOSTIC`
- **MAY** leave entities without an `entity_category` ŌĆö they then land in the default section (sensors, actuators, ŌĆ”) without special grouping

### Coordinator binding

- **MUST** bind entities to exactly one coordinator (via the base class that subclasses `CoordinatorEntity`)
- **MUST** read the coordinator from `entry.runtime_data.coordinators["<role>"]` in the platform `async_setup_entry` (see `ha/runtime-data-pattern` and `ha/coordinator-patterns`)
- **SHOULD** bind entities that combine data from multiple coordinators to the **primary coordinator** and read other coordinator data on-demand via `runtime_data` ŌĆö HA does not directly support multi-coordinator subscriptions
- **MUST NOT** run blocking I/O inside `_handle_coordinator_update`; the callback runs on the event loop and must work synchronously with the coordinator data

## Acceptance Criteria

- [ ] `entity.py` contains a cross-platform base entity class with `_attr_has_entity_name = True` and `_attr_device_info` assignment
- [ ] Every platform entity subclasses the base class
- [ ] Every entity has a `translation_key` (either `_attr_translation_key` or via `EntityDescription`)
- [ ] Every entity has a `unique_id` in the format `{entry_id}_{resource_type}_{resource_slug}_{suffix}`
- [ ] No code in `custom_components/<domain>/` sets `self.entity_id = "..."`
- [ ] Platforms with > 2 datapoints use `EntityDescription` tuple lists
- [ ] Entities that change configuration are marked as `EntityCategory.CONFIG`
- [ ] Entities that surface read-only diagnostic info are marked as `EntityCategory.DIAGNOSTIC`
- [ ] `_handle_coordinator_update` contains no blocking I/O
- [ ] Quality scale markers: **Bronze** for stable `unique_id`, additional **Silver** for `_attr_has_entity_name = True` + `translation_key` + `EntityDescription` pattern

## Open Questions

- **Platform threshold for the `EntityDescription` requirement**: "more than two datapoints" is a heuristic. Should the threshold be 1 (always) or 5 (more lenient)? `kamerplanter-ha` actually uses the pattern from the first datapoint per platform.
- **Suggested-object-ID ban**: `_attr_suggested_object_id` is currently formulated as MAY. Should it become a full ban because it makes slug behaviour unpredictable?
- **Translation keys for sub-states**: `kamerplanter-ha` defines `state.<value>` mappings per sensor translation key (for example phase values). Does this convention belong here or in `ha/translations`?
- **Multi-coordinator subscriptions**: HA has experimental APIs for multi-coordinator listening. Should the spec allow them once stable, or does single-coordinator binding remain mandatory?
- **`RestoreEntity` threshold**: When does the spec require the `RestoreEntity` mixin (state persistence across HA restart)? Currently a non-goal; a follow-up spec materialises once the first concrete entity class needs it.
