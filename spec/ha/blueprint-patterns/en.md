# HA Blueprint: Authoring Patterns

Status: draft

## Context

A Home Assistant **blueprint** is an automation, script, or template-entity configuration with selected parts deliberately left blank for the user to fill in without code ([HA docs: Blueprints](https://www.home-assistant.io/docs/blueprint/)). Exactly three domains are supported: `automation`, `script`, and `template`. Unlike a custom integration (Python, coordinator, config flow), a blueprint is a pure **YAML artifact** with no runtime code — all of its quality is decided by schema correctness, the UX of its input fields (selectors), and template robustness.

A blueprint lives as a `.yaml` file under `<config>/blueprints/<domain>/<author>/<file>.yaml` ([HA docs: Tutorial](https://www.home-assistant.io/docs/blueprint/tutorial/), [Using Blueprints](https://www.home-assistant.io/docs/automation/using_blueprints/)). The top-level `blueprint:` key requires only `name` and `domain`; everything else (`description`, `author`, `homeassistant.min_version`, `input`, `source_url`) is optional ([HA docs: Schema](https://www.home-assistant.io/docs/blueprint/schema/)). Concrete values are made referenceable through the custom YAML tag `!input <name>`; every value referenced this way must be declared in the `input:` map.

The central lever for reusability is **selectors**: they define both the accepted values of an input and how that input is rendered in the HA UI — "having good selectors makes a blueprint easier to use from the UI" ([HA docs: Selectors](https://www.home-assistant.io/docs/blueprint/selectors/)). Imported blueprints remember their `source_url` so HA can later update them from their origin.

This spec distills the official HA blueprint documentation (as of 2024–2026) into a binding authoring convention for every blueprint generated by skills or agents in this plugin. Because blueprints have no Quality-Scale counterpart (the Quality Scale applies to integrations), the quality bar here is the official documentation plus the established "Share your Blueprints" forum conventions.

## Goals

- Pin down the `blueprint:` header and the file layout (`blueprints/<domain>/<author>/<file>.yaml`) as a binding structure
- Enforce selectors as the primary UX surface: every input carries a type-appropriate selector instead of a bare free-text field
- Prefer `target`/`entity` selectors over `device` selectors so blueprints work across devices
- Wire the templating path correctly: never reference `!input` directly in templates, bridge it through `variables:` or `trigger_variables:`
- Require a deliberate `mode` (with `max` for `queued`/`parallel`) instead of the silent `single` default where the automation needs it
- Establish robustness against `unavailable`/`unknown` states and sensible defaults as a duty
- Pin down versioning and distribution discipline (`source_url`, backward-compatible updates, My Home Assistant import link)

## Non-Goals

- Custom-integration patterns (coordinator, config flow, entity architecture) — covered by the other `ha/` specs
- HACS distribution of blueprints as a repository — its own follow-up spec once a concrete need arises
- A complete selector reference — this spec requires correct selector usage but does not replicate the exhaustive [selector table in the HA docs](https://www.home-assistant.io/docs/blueprint/selectors/)
- Authoring Jinja templates beyond the blueprint-specific `!input` bridge — general template hardening is HA-wide knowledge, not blueprint-specific
- Translation/localization of blueprint text — HA blueprints carry their text inline; an i18n mechanism does not (yet) exist

## Requirements

### Blueprint header and file layout

- **MUST** carry the top-level `blueprint:` key with at least `name` and `domain`; `domain` is exactly one of `automation`, `script`, `template`
- **MUST** place the file with a `.yaml` extension under `<config>/blueprints/<domain>/<author>/<file>.yaml` — `<author>` is the author/namespace folder, `<domain>` matches the `domain` key
- **MUST** set a `description` that explains, in complete sentences, what the blueprint does and which inputs it expects — it is the only documentation the user sees on import
- **SHOULD** phrase `description` as a Markdown multi-line block (YAML `>` or `|`) and keep it understandable for **non-technical users** (no internal jargon, no entity IDs in prose)
- **SHOULD** set `source_url` to the canonical origin (forum post or GitHub raw URL) once the blueprint is shared — HA stores it for later updates
- **SHOULD** set `homeassistant: { min_version: <YYYY.M.0> }` when the blueprint uses features that require a minimum HA version (see sections: `2024.6.0`)
- **MAY** set `author`; if absent, the namespace folder acts as the de-facto author marker
- **MUST NOT** nest executable configuration (triggers, conditions, actions, sequence) inside the `blueprint:` block — those sit at the file's top level alongside `blueprint:`

### Inputs

- **MUST** declare every value referenced via `!input <name>` under `blueprint.input.<name>` — an `!input` without a matching declaration breaks the import
- **MUST** give each input at least a `name` (UI label) and a `selector`; `description` is strongly recommended
- **MUST** treat an input without a `default` as a **required** field; a set `default` makes it optional (the user may leave it blank)
- **SHOULD** set a sensible `default` for optional behavior extensions (e.g. "additional action") — typically `default: []` for action/entity lists, so the blueprint stays runnable without being filled in
- **SHOULD** group related inputs into **sections**: a section is an input entry that itself contains a nested `input:` key, and supports `collapsed: true` to collapse by default
- **MUST** set `homeassistant.min_version: 2024.6.0` (or higher) when using sections — sections are only available from that version onward
- **MUST NOT** reuse the same input name twice or declare an input that is never referenced via `!input` (dead input)

### Selectors

- **MUST** give each input a domain-appropriate selector (`entity`, `target`, `device`, `area`, `number`, `boolean`, `time`, `action`, `text`, `select`, `object`, …) instead of a bare free-text field
- **SHOULD** **prefer** a `target` or `entity` selector over a `device` selector for choosing the devices/entities to control — `target`/`entity` work across entities and do not break when the user uses a different device model
- **MUST** filter selectors where sensible: narrow `entity` selectors by `domain`, `device_class`, `integration`, or `supported_features` so the user is only offered matching values (e.g. `domain: binary_sensor`, `device_class: motion`)
- **SHOULD** set `multiple: true` when the input should hold several values — the delivered value is then a **list** instead of a single string; actions/templates must account for this
- **SHOULD** set `min`, `max`, `step` and — where sensible — `unit_of_measurement` and `mode` (`slider`/`box`) on `number` selectors so the UI renders sensible bounds
- **MUST NOT** use a `device` selector merely to resolve its entities by template afterward when a direct `entity`/`target` selector achieves the same without template fragility

### Templating bridge (`variables` / `trigger_variables`)

- **MUST** first map an `!input` value needed in a **template** (Jinja) onto a named variable via a `variables:` block (script level) — `!input` is referenceable outside the blueprint metadata but is **not** directly available as a template variable
- **MUST** use the automation-level `trigger_variables:` block for `!input` values needed in **trigger configuration** (e.g. a templated `value_template`, `for`) — it exists precisely for this purpose (making blueprint inputs available in triggers)
- **MUST** accept that `trigger_variables` only allow **limited templates** and do **not** re-apply when the template value changes — do not place logic there that relies on runtime re-evaluation
- **SHOULD** deliberately distinguish `variables` (per-trigger/action, reacts on trigger match) from `trigger_variables` (automation level, evaluated once at setup time) — they are separate variable types with different lifecycles
- **MUST NOT** write `!input` directly into a Jinja expression (`{{ !input foo }}`) — it is syntactically invalid and the most common import breakage

### Automation-specific: mode, trace, id

- **MUST** choose `mode` deliberately — `single` (default), `restart`, `queued`, or `parallel`; the choice is justified by the expected trigger behavior, not left to the silent default
- **MUST** explicitly set `max` for `mode: queued` or `mode: parallel`, or deliberately accept the default of `10` — `max` caps runs executing and/or queued at once
- **SHOULD** choose `restart` for "latest trigger wins" logic (e.g. motion light with off-delay) and `single` for non-reentrant flows
- **MAY** give the generated automation a stable `id` so trace and UI references stay stable across re-imports
- **SHOULD** be structured so the HA **trace** view stays meaningful (named steps, no monolithic template action in which all logic disappears)

### Script and template blueprints

- **MUST** put executable logic under `sequence:` for `domain: script` (instead of `trigger`/`action`) — script blueprints have no triggers
- **MUST** structure the template-entity configuration per the relevant template platform for `domain: template` (e.g. `sensor:`, `binary_sensor:`) — template blueprints define entities, not automations
- **MUST NOT** use automation-specific keys (`mode`, `trigger`, `trigger_variables`) in script or template blueprints where the domain does not know them
- **MAY** reuse the same input/selector/`variables` patterns as an automation blueprint — the input layer is identical across domains

### Robustness and UX

- **MUST** guard templates against `unavailable`/`unknown`/`None` states (e.g. check `states('sensor.x') not in ['unavailable','unknown']`, use the `default` filter in Jinja) before accessing the value
- **MUST NOT** hard-code entity IDs, area names, or other installation-specific values — everything configurable goes through `!input` with a selector
- **SHOULD** choose sensible defaults so the blueprint is immediately runnable with minimal filling-in (required fields only)
- **SHOULD** phrase conditions defensively so that repeated trigger firings stay idempotent (no double-switching, no oscillation)
- **MUST** produce valid YAML that passes HA's blueprint schema validation — a blueprint that fails on import is worthless

### Distribution and versioning

- **SHOULD** provide a **My Home Assistant import link** (`https://my.home-assistant.io/redirect/blueprint_import/?blueprint_url=<url>`) when sharing — it opens the import dialog pre-filled with the blueprint via a single URL
- **MUST** stay **backward-compatible** when updating an already-published blueprint: do **not** rename or remove existing selectors and input names; **newly** added inputs/selectors **must** carry a `default` so existing automations keep working
- **SHOULD** create a **new** blueprint for substantial, non-backward-compatible changes instead of breaking the existing one
- **MAY** carry a version note in `description` or as a comment — HA has no dedicated blueprint version field
- **SHOULD** be aware that a re-import updates content (e.g. `description`) but does **not** update the stored `source_url` when the source is changed in the YAML (known core behavior, [core#123025](https://github.com/home-assistant/core/issues/123025)) — therefore choose a stable canonical `source_url` from the outset

## Acceptance Criteria

- [ ] `blueprint:` contains `name` and `domain`; `domain` ∈ {`automation`, `script`, `template`}
- [ ] File lives under `blueprints/<domain>/<author>/<file>.yaml` with a `.yaml` extension
- [ ] `description` explains purpose and inputs in complete, non-technical sentences
- [ ] Every `!input` reference has a corresponding `blueprint.input.<name>` declaration; no dead or duplicate input
- [ ] Every input carries a type-appropriate `selector`; no bare free-text field for structured values
- [ ] Device/entity selection uses `target`/`entity` instead of `device` where cross-entity function is needed
- [ ] Selectors are filtered (`domain`/`device_class`/`integration`) where it sensibly narrows the choice
- [ ] `multiple: true` inputs are treated as lists in actions/templates
- [ ] No `!input` appears directly in a Jinja template; templating goes through `variables` or `trigger_variables`
- [ ] Sections (if used) come with `homeassistant.min_version: 2024.6.0`
- [ ] `mode` is set deliberately; for `queued`/`parallel`, `max` is explicit or deliberately left at the default
- [ ] Templates are guarded against `unavailable`/`unknown`; no hard-coded installation-specific values
- [ ] Newly added inputs (on updates) carry `default`; existing input names/selectors are unchanged
- [ ] The blueprint passes HA's schema validation (import or `hass --script check_config` locally)

## Open Questions

- **Validation toolchain**: Which locally runnable check most reliably verifies a blueprint without a running HA instance? Candidates: a YAML schema lint, `hass --script check_config` in a container, or a dedicated blueprint schema validator. Until resolved, importing into an HA instance is the authoritative gate.
- **Reference blueprint in the portfolio**: Unlike the integration specs (grounded in `nolte/kamerplanter-ha`), this spec has no portfolio-internal reference blueprint yet. Once one exists, its concrete patterns should be anchored here as an example.
- **Template-blueprint depth**: The template domain is only roughly delimited here. Once a real template blueprint exists, a dedicated `ha/template-blueprint` spec with platform-specific requirements may be needed.
- **Sections minimum version vs. audience**: Sections require `2024.6.0`. Should the spec recommend sections generally (UX gain) or stay conservative so blueprints also serve older HA installations?
- **Idempotency heuristic**: "Idempotent on repeated triggers" is phrased as a SHOULD. A checkable heuristic (which condition patterns count as sufficient) is still missing.
