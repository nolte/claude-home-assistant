# Parallel Working Copies

Status: draft
Implementation: documentary-only—the MUSTs in this spec are contributor-behaviour conventions (worktree paths, lifecycle, uncommitted-change transfer, Claude Code session scoping). The §Notes-on-coverage paragraph already lists which MUSTs have post-hoc-observable Acceptance Criteria and which are convention-only; no callable Claude Code skill or agent in this plugin enforces the rules. Enforcement is by human and AI-agent practice, with `git worktree list --porcelain` checks supporting the observable subset of ACs. The single exception is the §Branch-to-worktree mapping MUST that the primary checkout stays on `develop`: a pre-commit guard (`scripts/guard_primary_checkout.sh`, the `guard-primary-checkout` hook) enforces it mechanically at commit time in the primary checkout, so that one rule isn't purely documentary.

## Context
A single primary checkout of a repository can hold only one branch at a time. As soon as a contributor (human or AI agent) wants to advance two or more feature branches in parallel (for example, drafting a spec on `feat/parallel-working-copies` while a long-running build runs on `feat/mermaid-diagrams`), switching branches in place destroys the working tree of whichever branch is paused: uncommitted edits collide, build outputs invalidate, IDE indexes thrash, and any tooling that has cached the cwd (Claude Code sessions, language servers, watchers) is forced to re-bootstrap.

`git worktree` solves this without cloning: it adds a second working tree that shares the same `.git` object database but has its own independent index, working files, and `HEAD`. This spec defines the conventions under which the portfolio uses worktrees so that parallel feature work is reliable, auditable, and consistent with the existing branching, project-structure, and pull-request specs.

Readers: contributors (human and AI agents) doing parallel feature work in this portfolio's repositories, plus reviewers who verify worktree hygiene before merge.

## Goals
- Allow a contributor to advance two or more feature branches in parallel without the working tree of one branch overwriting another
- Provide stable conventions for worktree paths, branch-to-worktree mapping, and lifecycle (create, use, retire) so the layout is predictable across repositories and contributors
- Define how Claude Code sessions are scoped to worktrees so `CLAUDE.md` resolution, current-working-directory assumptions, and plugin loading remain coherent
- Guarantee that interrupted feature work in a worktree is recoverable: a foundational implementation plan is on disk before substantive work begins, and that work runs in a fresh top-level session that `claude --resume` can reopen
- Reserve the primary checkout exclusively for the integration role (sitting on `develop`) so that it remains a stable launchpad from which every new feature worktree branches off, and stays continuously available for integration tasks (rebases, conflict resolution, release inspection); the operative rule is the MUST in Requirements §Branch-to-worktree mapping

## Non-Goals
- Branch naming and protection rules—defined in `spec/project/branching-model/`
- Pull-request creation, review, and merge mechanics—defined by the `pull-request-create` / `pull-request-merge` skills
- Release automation, version bumping, and `main` refresh—defined in `spec/project/branching-model/` and the portfolio's release automation
- Repository scaffolding (`Taskfile.yml`, `.github/`, MkDocs)—a project-structure concern outside this spec
- IDE-specific multi-root workspaces, devcontainer setups, or remote-development tooling
- Resolution of merge conflicts between feature branches—a normal `git merge` / `git rebase` concern, unaffected by whether the work happened in a worktree
- CI runners and ephemeral cloud development environments that clone a fresh workspace per run—those start from a clean state by construction and don't require worktree conventions; this spec governs persistent local working copies

## Requirements

### Path layout
- **MUST** place every additional worktree outside the primary checkout's directory tree; a worktree path **MUST NOT** be nested under the primary repository directory
- **SHOULD** place worktrees under a centralized, per-machine-configurable layout `${NOLTE_WORKTREE_ROOT:-~/repos/.worktrees}/<repo>/<short-slug>/`, where the root is read from the `NOLTE_WORKTREE_ROOT` environment variable—which each contributor chooses freely for their machine—and defaults to `~/repos/.worktrees` when the variable is unset, `<repo>` matches the repository name from the `origin` remote, and `<short-slug>` is a kebab-case abbreviation of the feature (not necessarily identical to the branch name). The reference helper `task worktree:add -- <branch> [slug]` (`scripts/worktree_add.sh`) is the conformant creation path: it resolves the root from `NOLTE_WORKTREE_ROOT` (same default), derives `<repo>` from the `origin` remote, and creates the branch off `origin/develop` with an explicit base ref per §Lifecycle: Create. A single configurable root keeps every parallel working copy on the machine in one predictable place, reduces the chance of accidentally nesting a worktree under another repository, and makes orphan worktrees easy to spot with a single `ls "${NOLTE_WORKTREE_ROOT:-~/repos/.worktrees}/<repo>/"`
- **MAY** instead name worktrees as a sibling of the primary checkout in the form `<repo>-<short-slug>/` (for example, the primary checkout `~/repos/github/claude-home-assistant/` plus the worktree `~/repos/github/claude-home-assistant-mermaid/`) when a contributor prefers per-repository locality over a single centralized root; mixing both layouts within one machine is permitted but **SHOULD** be avoided per repository
- **MUST NOT** place a worktree path inside another repository's working tree, inside `node_modules/`, inside `.venv/`, inside `.claude/` (a Claude Code plugin directory that plugin-management and reload operations may rewrite wholesale; the historical `.claude/worktrees/<slug>/` pattern is the concrete drift this clause forbids), inside `.git/` (the git administrative database), or inside any other directory that a tool may delete or rewrite wholesale
- **MUST NOT** add a `.gitignore` entry whose effect is to hide a spec-violating worktree path from `git status` (for example, `.claude/worktrees/` or a bare `worktrees/`); nested-worktree drift **MUST** remain surfaced by `git status` so a contributor or reviewer notices it the first time it appears, rather than silently accumulating

### Branch-to-worktree mapping
- **MUST** check out exactly one branch per worktree; a worktree **MUST NOT** be left in a detached-HEAD state for ongoing feature work (detached HEAD is acceptable only for short-lived inspection tasks such as `git worktree add --detach` for a bisect)
- **MUST** ensure that no branch is checked out in two locations simultaneously; the primary checkout and any worktree together **MUST** present each branch at most once (this is a property `git` enforces—the spec restates it because violations indicate a process error, not a tooling error)
- **MUST** keep the primary checkout on `develop` at all times; the primary checkout serves as a stable launchpad from which new feature worktrees branch off and stays continuously available for integration tasks (rebases, conflict resolution, release inspection) without forcing a worktree teardown
- **MUST** perform every change to requirements, fixes, and extensions in a dedicated worktree per §Lifecycle: Create; the primary checkout **MUST NOT** be switched off `develop` onto a feature branch, even when only one feature is in flight, because a single in-place branch switch destroys the launchpad role and invites the very nested-worktree and `.audits/`-drift failure modes the spec exists to prevent
- **MUST** perform every branch-mutating operation on a branch a worktree owns—merging or rebasing `origin/develop` into it to clear a branch-freshness lag, resetting it, amending its tip, or otherwise re-pointing the ref—from inside the owning worktree (for example `git -C <worktree> merge origin/develop`), never by issuing `git switch -C <branch>`, `git branch -f <branch>`, or any equivalent ref-moving command from the primary checkout or a sibling working copy. Moving the ref from elsewhere slides it out from under the owning worktree, whose index and working tree stay at the old tip while `git worktree list` already reports the new one; the silent symptom is a worktree that shows the very files the operation introduced as uncommitted "changes" it never made, and the only safe recovery is `git worktree remove --force` after confirming the worktree holds nothing the branch tip doesn't already capture. This rule is the ref-level companion to the preceding two: the primary checkout stays on `develop` not only by not checking out a feature branch, but by not mutating one a worktree owns
- **SHOULD** wire a pre-commit guard in each repository that mechanically enforces the two primary-checkout MUSTs above in the primary checkout; the reference implementation is `scripts/guard_primary_checkout.sh`, registered as the `guard-primary-checkout` hook in `.pre-commit-config.yaml`. It blocks any commit made in the primary checkout while it's off `develop`, and stays inert inside linked worktrees—which it distinguishes by comparing `git rev-parse --git-dir` against `--git-common-dir` (equal only in the primary checkout). This guard is the one mechanical exception to this spec's `Implementation: documentary-only` posture: it closes the commit-time failure mode—feature work committed directly in the primary checkout—that the post-hoc §Acceptance Criteria can only detect after the fact
- **SHOULD**, when migrating a repository that historically held a long-running feature branch in the primary checkout, transfer that feature into a new worktree via `git worktree add -b <existing-branch> <worktree-path>` (re-pointing the branch to the worktree) and reset the primary checkout to `origin/develop` before starting any further feature work
- This spec sets no maximum on concurrent worktrees; contributors **SHOULD** keep the count to a reviewable number and rely on the centralized worktree root (`${NOLTE_WORKTREE_ROOT:-~/repos/.worktrees}/<repo>/`, per §Path layout) for orphan visibility. Review-throughput limits are out of scope for this spec

### Lifecycle: Create
- **MUST** create new worktrees with `git worktree add -b <branch> <path> <base-ref>` so the branch is created as part of the worktree command and the base ref is explicit
- **SHOULD** use `origin/develop` (after `git fetch origin develop`) as the base ref rather than the local `develop`, so the worktree starts from the remote tip and the primary checkout's local `develop` is irrelevant to the worktree's starting point
- **MUST** follow the branch-prefix rules from `spec/project/branching-model/` (`feat/`, `fix/`, `chore/`, `docs/`, `exp/`); the worktree's path slug **MAY** drop the prefix for brevity but the branch itself **MUST NOT**

### Lifecycle: Plan before work
- **MUST**, before any substantive content work begins in a freshly created worktree (editing specs, skills, agents, docs, or code on its branch), record a foundational implementation plan on disk inside that worktree; the plan is written before the work it governs, so a fresh session—or a contributor returning after a crash—can pick the work up from a known starting point rather than reconstructing intent from a half-finished diff
- **SHOULD** store the plan at `.resume/<slug>/plan.md` (where `<slug>` is the worktree's short slug per §Path layout), under `.resume/`; this path is gitignored, so the plan is a worktree-local working aid rather than a committed artefact, and it never competes with the feature's real changes for review attention
- **SHOULD** give the plan the shape this kind of document uses: a goal statement, the researched current state, the load-bearing design decision (with the open questions to confirm before work starts), the ordered work steps, the invariants and guardrails carried over from `CLAUDE.md` and the governing specs, and a status/resume-anchor checklist whose first unchecked box is where the next session resumes
- **SHOULD** let the conformant creation helper seed the plan: `task worktree:add -- <branch> [slug]` (`scripts/worktree_add.sh`) writes a `.resume/<slug>/plan.md` stub so the gate is self-serving, and prints a reminder that the stub **MUST** be filled in before substantive work begins
- This gate is convention-driven, consistent with this spec's `Implementation: documentary-only` posture: no hook hard-aborts work when the plan is absent. The stub-writing helper makes authoring the plan the path of least resistance rather than a blocking check; a mechanical pre-work guard is deferred on the same terms as the deferred `.audits/` removal guard under §Audit artefacts in multiple worktrees

### Lifecycle: Retire
- **MUST** remove a worktree with `git worktree remove <path>` once its pull request is merged or abandoned; deleting the directory with `rm -rf` is forbidden because it leaves the worktree registered in `.git/worktrees/` and produces "missing" entries in `git worktree list`
- **MUST** delete the local branch (`git branch -d <branch>` or `-D` if the branch was merged via squash) after the worktree is removed, so stale local branches don't accumulate
- **SHOULD** run `git worktree prune` periodically (or after a forced removal) to reap administrative entries for worktrees whose directories disappeared
- **MUST NOT** retire the primary checkout via `git worktree remove`; the primary checkout is the linked-worktree root, not a removable worktree

### Harness-initiated and agent-initiated worktrees
A Claude Code session can create a worktree programmatically—most prominently via the harness `Agent({isolation: "worktree"})` option, but also via any future skill or agent that wraps `git worktree add`. These programmatic worktrees are bound by the same rules as contributor-initiated ones, with the following additions to close the historical drift in which the harness defaulted to `<primary-checkout>/.claude/worktrees/<slug>/`:

- **MUST** apply every rule from §Path layout to harness-initiated and agent-initiated worktrees without exception; the §Path layout MUST-NOTs (including the explicit `.claude/` clause and the `.gitignore`-concealment clause) bind the harness and any agent identically to a human contributor
- **MUST NOT** allow a harness, skill, or agent to materialize a worktree at a path that violates §Path layout; if the harness's built-in default points to such a path (for example, `<primary-checkout>/.claude/worktrees/agent-<hash>/`), the contributor **MUST** override the default—by setting the documented worktree-root environment variable (such as `CLAUDE_AGENT_WORKTREE_ROOT`) or the equivalent Claude Code settings hook—to a spec-conformant root before the first agent invocation that uses `isolation: "worktree"`
- **SHOULD** point the harness-/agent-worktree root at `${NOLTE_WORKTREE_ROOT:-~/repos/.worktrees}/<repo>/agents/<slug>/`, so that automatically created worktrees stay spatially separable from contributor-initiated worktrees under `${NOLTE_WORKTREE_ROOT:-~/repos/.worktrees}/<repo>/<slug>/` and orphan inventories remain readable with a single `ls`; this keeps the harness-/agent root (`CLAUDE_AGENT_WORKTREE_ROOT` per the bullet above) nested under the same per-machine-configurable root that §Path layout defines for contributor worktrees
- **MUST** retire harness-/agent-initiated worktrees automatically at the end of the agent's task when no changes need to be preserved (the documented harness default: "the worktree is automatically cleaned up if the agent makes no changes"); contributors **MUST NOT** disable this automatic cleanup
- **MUST** retire a harness-/agent-initiated worktree manually via `git worktree remove <path>` once its changes have been merged or transferred per §Uncommitted changes between worktrees; `rm -rf` is forbidden for the same reason it's forbidden under §Lifecycle: Retire
- **MUST** pre-authorize, before launching a non-interactive agent that has to act inside a worktree (most prominently a background `Agent(run_in_background: true)` run, or any harness worktree whose prompts the operator can't answer), both of the following, because a non-interactive agent can't answer a permission prompt and any tool call that's neither allowed by the harness default nor pre-authorized is hard-denied, stalling the agent:
  1. the worktree root as an allowed working directory. The session's default permitted tree is the primary checkout plus the harness's transient scratch directories; the spec-conformant worktree root under `${NOLTE_WORKTREE_ROOT:-~/repos/.worktrees}/<repo>/` (per §Path layout) lies outside it, so it **MUST** be added via `/add-dir <worktree-root>` or the `additionalDirectories` settings key before the first such agent invocation.
  2. every command class the agent needs to mutate the worktree and the remote: at minimum `git` and `gh`, plus the project's build, test, and setup runner (for example `task`) and any fetch tooling the task requires (for example `curl`, `unzip`).
- **MUST** satisfy the command-class authorization above through session-scoped or developer-local configuration (the session `/permissions` grant or `.claude/settings.local.json`), and **MUST NOT** add it to the committed `.claude/settings.json`. The mutation-capable commands an autonomous agent needs (`git commit` / `git push`, `gh pr create`, task-runner wildcards) are exactly the pattern classes that have no place in the committed allowlist, so granting them there would violate the allowlist discipline; the working-directory grant from the preceding bullet is likewise a session or developer-local concern, never a committed-allowlist entry.
- **SHOULD** treat the interactive primary session as the fallback when pre-authorizing mutation commands for a background agent is undesirable: the operator approves the interactive session's `git` and `gh` prompts as they appear, so the work runs to a pushed branch and an opened PR without relaxing the committed allowlist. An agent restricted to read-only worktree access can still perform the analysis and the `Edit` / `Write` steps, with the `git` and `gh` steps run from the interactive session and any uncommitted edits transferred per §Uncommitted changes between worktrees.
- **SHOULD** prefer the `git -C <worktree> <subcommand>` form over `cd <worktree> && git …` for agent-issued git commands, because per-command authorization can deny the compound form while admitting the explicit-directory form, and because it sidesteps the shell-`cwd`-reset hazard noted under §Claude Code session scoping.

### Uncommitted changes between worktrees
- **MUST NOT** copy uncommitted changes between worktrees through the filesystem (`cp`, `rsync`, editor "save as"); each worktree's working tree is independent and a filesystem copy bypasses the git index and produces silently divergent state
- **MUST** transfer uncommitted work between worktrees through git: `git stash push` in the source worktree, then `git stash apply` (or `pop`) in the target worktree, or via a temporary commit pushed to a shared remote ref
- **SHOULD** prefer a temporary commit over `git stash` when the work needs to survive a long delay, so the change is durable on disk via a commit object rather than the stash reflog

### Claude Code session scoping
- **SHOULD** start one Claude Code session per worktree, launched from the worktree's root directory (`cd <worktree>; claude`) so that `CLAUDE.md` hierarchy resolution, the auto memory project namespace, and current-working-directory defaults all bind to the worktree
- **SHOULD** perform the substantive, long-running work of a worktree in a fresh top-level Claude Code session started from the worktree directory rather than dispatching it into a worktree-isolated subagent (`Agent({isolation: "worktree"})`) or a Workflow run, because only a top-level session's transcript is independently resumable: `task resume` (and `claude --resume` / `claude --continue`) lists and reopens top-level session transcripts, whereas a subagent's or Workflow agent's transcript lives under its parent session and can't be `claude --resume`d on its own. Pairing this with the §Lifecycle: Plan before work gate is what makes interrupted feature work recoverable—the plan on disk records where the work stands, and the resumable session reopens the context that produced it. A per-skill checkpoint format complements this session-level resume

- **MAY** continue using a single session across worktrees by passing absolute paths to file-reading tools and prefixing shell commands with `cd <worktree> && …`; this is supported but **SHOULD NOT** be the default, because the harness resets the shell `cwd` between bash invocations and `CLAUDE.md` resolution remains anchored to the original session's launch directory
<!-- vale Microsoft.Contractions = YES -->
- **MUST NOT** edit a file via a Claude Code session whose launch directory is a different worktree without using the absolute path, because relative-path tooling will resolve to the wrong working tree
- **MUST NOT** run `task plugin:reload` (or any equivalent plugin-development command from `claude-home-assistant`) in two sessions that target the same plugin source root simultaneously; each Claude Code session loads its own plugin image, but two sessions reloading the same source root concurrently produce races on file watchers and reload semantics

### Local state and untracked configuration
- **MUST** treat untracked, machine-local configuration files (`.env`, `.claude/settings.local.json`, IDE workspace files, build caches, `node_modules/`, `.venv/`) as worktree-local; each worktree maintains its own copy
- **MUST NOT** symlink machine-local configuration from a worktree back to the primary checkout or to another worktree; symlinks across worktrees create hidden coupling that breaks the isolation guarantee this spec exists to provide
- **SHOULD** rebuild language-specific environments (`uv sync`, `npm install`, `task setup`) inside each worktree on first use rather than reusing the primary checkout's `.venv/` or `node_modules/`

### Audit artefacts in multiple worktrees

Files under `.audits/` carry the persistent on-disk output of review, sweep, and lektorat skills (with a per-target plan format and a per-run snapshot pattern); they behave like any other working-tree content when produced inside a worktree rather than the primary checkout, with the following clarifications and conventions:

- Audit artefacts under `.audits/` are worktree-local: every audit run writes only into the working tree of the worktree it was launched from, including timestamped run-snapshot directories (`.audits/lektorat/<YYYY-MM-DD-HHMM>/`, `.audits/<skill>/<run>/`); files become visible to other worktrees only once the originating branch is merged
- This section is the disambiguation anchor for every audit, readiness, and freshness skill that scopes itself to "the working tree": on a multi-worktree machine, "the working tree under audit" **MUST** be read as the worktree the audit was launched from, never the primary checkout or a sibling worktree; downstream audit, readiness, and freshness specs **SHOULD** point here rather than restating the rule
- **MAY** commit audit files onto the worktree's feature branch—this is explicitly permitted, so per-item follow-up (check-off, `## Processing log` updates, the cleanup-commit defined in the review-plan lifecycle) happens inside the same PR context that produced the audit
- **MUST NOT** modify the same `.audits/<review-type>/<target>.md` path concurrently in two worktrees; the per-(review-type, target) uniqueness rule of the review-plan convention isn't observable across worktrees and parallel modifications produce merge conflicts or lost updates (timestamped run-snapshot directories are exempt, because their per-run paths are unique by construction)
- **SHOULD** run portfolio-wide audits that carry an "only one open" constraint (for example `skills-agents-sweep` with `status: open`) inside the primary checkout on `develop`, because the open-plan check only sees the working copy the skill started in—a second sweep launched from a parallel worktree would not notice the first
- **SHOULD** handle an uncommitted `.audits/` diff in a worktree deliberately before `git worktree remove <path>` by committing it onto the worktree's branch, transferring it per §Uncommitted changes between worktrees, or acknowledging it as discarded, so the audit trail doesn't silently disappear; the `rm -rf` prohibition from §Lifecycle: Retire holds regardless. A mechanical guard—a removal-time wrapper or hook that blocks or warns on an uncommitted `.audits/` diff—is deliberately deferred while this spec stays `Implementation: documentary-only`; it becomes worth building only once a discarded-diff incident is recorded or Claude Code exposes a pre-worktree-remove hook surface

### Interaction with other portfolio specs
- **MUST** continue to follow the repository's pull-request workflow for PR creation from a worktree; the worktree's branch is pushed and the PR opened identically to a primary-checkout branch
- **MUST** keep `spec/project/branching-model/`'s rule that `main` is presentation-only; a worktree **MUST NOT** be created against `main` for feature work
- **MUST** apply the project quality gate inside each worktree before opening or marking a PR ready, because the worktree has its own working tree and a green gate in a different worktree isn't evidence for this one

## Acceptance Criteria
- [ ] `git worktree list` lists the primary checkout plus one entry per active feature branch and no entries marked `prunable`
- [ ] No worktree path is nested under the primary checkout's directory
- [ ] No branch appears in more than one worktree-list entry
- [ ] The primary checkout is on `develop` at all times, regardless of how many features are in flight (verifiable with `git worktree list --porcelain`: the first entry's path equals the primary repository directory and its `branch` line reads `refs/heads/develop`)
- [ ] No feature branch (one whose name starts with `feat/`, `fix/`, `chore/`, `docs/`, or `exp/`) is checked out in the primary checkout; every feature branch present in `git worktree list --porcelain` is registered against a worktree path other than the primary checkout's path
- [ ] After a PR is merged, neither `git worktree list` nor `git branch --list` mentions the retired branch
- [ ] No worktree contains a symlink that resolves to a path inside another worktree or the primary checkout (excluding shared `.git` administrative links, which `git worktree` manages itself)
- [ ] Claude Code sessions opened against a worktree resolve the worktree's own `CLAUDE.md` (verifiable by inspecting the session's loaded project instructions)
- [ ] Each worktree carries its own `.venv/`, `node_modules/`, or other ecosystem-local environment when one is required to run the project's quality gate
- [ ] No active feature worktree (one whose registered branch starts with `feat/`, `fix/`, `chore/`, `docs/`, or `exp/`) is in a detached-HEAD state, as listed by `git worktree list --porcelain`
- [ ] No worktree's registered branch is `main`, as listed by `git worktree list --porcelain`
- [ ] Every PR opened from a worktree-originated branch shows a passing project quality-gate check on its own branch tip (verified in the PR's checks tab, not inherited from another worktree)
- [ ] The primary checkout is still listed as the root entry in `git worktree list` (its path matches the primary repository directory and it carries no `bare` annotation), so that the §Lifecycle: Retire `MUST NOT` against removing the primary checkout hasn't been violated
- [ ] Neither the root `.gitignore` nor any nested `.gitignore` contains an entry whose effect is to suppress a worktree-bearing directory from `git status` (concretely, no line matching `.claude/worktrees/`, `worktrees/`, or any equivalent path that would mask the historical nested-worktree drift)
- [ ] For every entry in `git worktree list --porcelain`, the registered `worktree` path lies outside the primary checkout's directory tree and outside any other repository's working tree on the same machine (verifiable with a one-shot `git worktree list --porcelain | awk '/^worktree /{print $2}'` plus a path-prefix check against the primary repository directory)
- [ ] No `.audits/skills-agents-sweep/*.md` file with `status: open` (or, by analogy, any future audit artefact carrying the same "only one open" lifecycle constraint under its own audit path) was introduced by a commit unreachable from `develop` (verifiable with `git log --all -- .audits/skills-agents-sweep/` plus a `git merge-base --is-ancestor` check against `develop`, applied analogously to any such future path), so the constraint remains observable on the primary checkout even when feature work happens in parallel worktrees

Notes on coverage: Enforcement of the `MUST follow the pull-request workflow` requirement under §Interaction with other portfolio specs is delegated to the pull-request workflow's own acceptance criteria and is intentionally not duplicated here. The `MUST remove a worktree with git worktree remove` retirement rule is tested authoritatively by AC5 (post-merge, no `git worktree list` or `git branch --list` mention); AC1's `prunable` clause is a supplemental hygiene check that also exercises the SHOULD-level `git worktree prune` rule. The MUST at §Lifecycle: Create requiring `git worktree add -b <branch> <path> <base-ref>` with an explicit base ref is a creation-time convention with no stable post-hoc observable; it's enforced by contributor practice rather than a mechanical AC. The MUSTs at §Uncommitted changes between worktrees (no filesystem copy of uncommitted changes; transfer only via `git stash` or a temporary commit) are likewise contributor-behaviour conventions, analogous to the dual-`task plugin:reload` prohibition under §Claude Code session scoping; no stable post-hoc observable AC is provided. The §Path layout `MUST NOT` clauses against nesting under `.claude/` and against `.gitignore`-concealment are covered by the two new ACs (the `.gitignore`-content check and the `git worktree list --porcelain` path-prefix check); the §Harness-initiated and agent-initiated worktrees MUSTs share that coverage for the path-layout half, while the automatic-cleanup MUST is a harness-default convention with no stable post-hoc observable (the absence of a stale worktree directory after an agent run is a transient property, not a property of the working copy under audit). The MUST NOT at §Audit artefacts in multiple worktrees against concurrent same-path modification across worktrees is a contributor-behaviour convention analogous to §Uncommitted changes between worktrees and §Claude Code session scoping; the two SHOULDs in that section (run portfolio-wide "only one open" audits from the primary checkout; handle an uncommitted `.audits/` diff deliberately before `git worktree remove`) are likewise convention-driven, with the new AC against open `skills-agents-sweep` reports outside `develop`'s lineage covering only the first of them. The MAY permitting commits of audit files onto a worktree's feature branch is a clarification, not a constraint, and carries no AC by construction. The two new MUSTs at §Branch-to-worktree mapping (primary checkout always on `develop`; all feature work in a dedicated worktree) are covered by the two new ACs against `git worktree list --porcelain`: the first AC asserts the primary checkout's branch line, and the second AC asserts that no feature-prefixed branch occupies the primary checkout. The new MUST at §Branch-to-worktree mapping forbidding ref-level mutation of a worktree-owned branch from another working copy is a contributor-behaviour convention analogous to §Uncommitted changes between worktrees; the resulting index/working-tree mismatch is a transient state with no stable post-hoc observable, so no AC is provided. The SHOULD-level migration clause is a one-shot transition rule with no stable post-hoc observable once the transition is complete. The new MUSTs at §Harness-initiated and agent-initiated worktrees requiring pre-authorization of the worktree-root working directory and the agent's command classes carry no committed-working-tree observable, so they're convention-driven and cross-referenced rather than AC-bearing here: the session `/permissions` grant and `.claude/settings.local.json` aren't committed, and the companion `MUST NOT` against placing mutation patterns in the committed `.claude/settings.json` is enforced by the allowlist discipline itself, not duplicated here.
