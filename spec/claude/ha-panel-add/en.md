# Skill: `ha-panel-add`

Status: draft

## Context

`ha/lovelace-views-panels` defines two larger frontend surfaces beyond the single card: the **custom view** (overrides the default masonry layout, renders core-supplied cards/badges in a custom layout) and the **custom panel** (a full-page page, linked from the sidebar, with real-time access to the Home Assistant object; core examples: dashboards, Map, Logbook, History). Both are registered as a custom element; the render framework is free (Lit, Preact, etc.), only React is explicitly excluded. This skill operationalizes the **custom-panel** part of the spec — not the custom view. Quality-scale marker: custom panels are **not part of the HA quality scale**; the pattern lives outside the scale.

This skill augments **one** custom panel into an **existing** integration or frontend repo: the panel custom element (receives `hass`, `narrow`, `route`, and `panel` as properties, registered via `customElements.define(...)`), the JS-module wiring, and the registration through `panel_custom` in `configuration.yaml` (`url_path`, `module_url`, optional `sidebar_title`/`sidebar_icon`/`config`/`embed_iframe`) — conformant to `ha/lovelace-views-panels`. Before generating it cleanly distinguishes the custom panel from a panel-mode view (single-card layout in the dashboard) and from the custom view as a layout container.

## Scope

Augmenting exactly one custom panel per run into an existing repo: the panel custom element (Lit or another non-React framework) with the property set HA supplies, the ES-module file, and the `panel_custom` entry in `configuration.yaml` with a unique `url_path` and `module_url`, optional `sidebar_title`/`sidebar_icon`/`config`/`embed_iframe`, and — when needed — the ES5-adapter wiring. The skill reads `ha/lovelace-views-panels` and validates offline.

## Goals

- Augment a custom panel spec-conformantly from a described full-page sidebar page, delimited from the custom view and the panel-mode view
- Establish the panel as a custom element (not React) with the property set HA supplies (`hass`, `narrow`, `route`, `panel`) and register it through `customElements.define(...)`
- Enforce the `hass` property channel as the only access to the HA state — no access outside `hass`
- Establish registration through `panel_custom` in `configuration.yaml` as the canonical entry point with a unique `url_path` and `module_url`
- Make the JavaScript-version choice (latest over ES5, the ES5 adapter only when needed) and the `embed_iframe` hint explicit

## Non-Goals

- The custom view as a layout container (overrides masonry, renders core cards/badges via `ll-*` events) — covered in `ha/lovelace-views-panels` (view part), not generated by this skill
- A single card or a panel-mode view (single-card layout in the dashboard) — `ha-lovelace-card-scaffold` / `ha/lovelace-card-patterns`
- Programmatic dashboard generation (strategies that compute whole views/dashboards) — `ha/lovelace-strategies`
- WebSocket commands the panel calls — `ha/frontend-websocket-commands`
- Greenfield scaffolding of an integration — `ha-integration-scaffold`
- Build stacks (Vite, esbuild, Rollup), TypeScript migration, and theming by the panel itself — separate follow-up specs

## Requirements

### Activation triggers

- **MUST** activate on phrasings like:
  - "add a custom panel", "register a sidebar panel", "create a full-page custom panel"
  - "expose this as a full-page page in the sidebar"
  - "füge ein Custom-Panel hinzu", "registriere ein Sidebar-Panel"

### Inputs

- **MUST** capture: `target_dir` (repo root) and the description of the full-page page (prose), from which the skill derives the `url_path` and panel name
- **MAY** capture: `url_path`, `sidebar_title`/`sidebar_icon`, a `config` block (available at runtime as `panel.config`), whether ES5 support is needed, and whether `embed_iframe` should be set

### Pre-flight (in order — abort on first failure)

- **MUST** check that `target_dir` is an existing repo and that the target location for `module_url` is resolvable (e.g. `www/`/`/local/` or a bundled frontend path)
- **MUST** run the delivery-shape check: if a custom view (layout inside the dashboard) or a panel-mode view (single card) covers the need, the skill **SHOULD** point that out; only a genuine need for a full-page page linked from the sidebar justifies the custom panel
- **MUST** read the `ha/lovelace-views-panels` spec
- **MUST NOT** overwrite an existing panel element, a `url_path`, or a `panel_custom` entry; on collision abort

### Generation rules (from `ha/lovelace-views-panels`)

- **MUST** define the panel as a custom element — the render framework is free (Lit, Preact, etc.), but it **MUST NOT** use React, since React is explicitly excluded
- **MUST** register the element through `customElements.define(...)`
- **MUST** accept the properties HA sets: `hass` (object, current HA state), `narrow` (boolean) and `panel` (object; config available as `panel.config`)
- **SHOULD** additionally accept `route` (object), which HA sets on the panel element
- **MUST NOT** access the HA state outside the `hass` property — `hass` is the real-time channel to the Home Assistant object
- **MUST** register the panel through the `panel_custom` component in `configuration.yaml`, with a unique `url_path` per entry and the panel module through `module_url` (ES module, e.g. `module_url: /local/example-panel.js`)
- **SHOULD** set `sidebar_title` and `sidebar_icon` so the panel is sensibly linked from the sidebar
- **MAY** pass arbitrary data to the panel through the `config` block (at runtime as `panel.config`) and **MAY** set `embed_iframe` when the panel should be served in an iframe instead of loaded directly into the frontend
- **SHOULD** ship without ES5 support as long as no wider browser support is required; **MUST** load the ES5 adapter before defining the element when ES5 support is required, via `window.loadES5Adapter().then(function() { customElements.define('my-panel', MyCustomPanel) })`
- **MUST** name identifiers per `ha/naming-conventions` and verify HA internals against the official docs (`ha/upstream-docs-verification`)

### Validation & report

- **MUST** validate offline: the panel is a custom element (not React) and accepts `hass`, `narrow`, `route`, `panel`; it is registered through `customElements.define(...)`; it does not access the HA state outside `hass`; the `panel_custom` entry in `configuration.yaml` has a unique `url_path` and `module_url`; ES5 support — when needed — loads the adapter before `customElements.define`
- **MUST** deliver a CONFORMANT / NEEDS-WORK report against the acceptance criteria from `ha/lovelace-views-panels` (panel part), plus the changed file paths and the quality-scale marker (**not part of the HA quality scale**)

### Prohibitions

- **MUST NOT** augment more than one panel per run
- **MUST NOT** generate a custom view, a card, or a strategy
- **MUST NOT** deploy to a running HA instance

## Acceptance criteria

- [ ] Skill runs the delivery-shape check (custom panel vs. custom view vs. panel-mode view) and augments only on a genuine full-page need
- [ ] The panel is a custom element (not React) and accepts `hass`, `narrow`, `route`, and `panel` (config via `panel.config`)
- [ ] The element is registered through `customElements.define(...)` and does not access the HA state outside `hass`
- [ ] The panel is registered in `configuration.yaml` through `panel_custom` with a unique `url_path`
- [ ] The panel module is referenced through `module_url` (ES module); `sidebar_title`/`sidebar_icon` set where sensible
- [ ] ES5 support — when needed — loads the ES5 adapter via `window.loadES5Adapter()` before `customElements.define`
- [ ] Report names the file paths and the quality-scale marker **not part of the HA quality scale**

## Open questions

- **View vs. panel as delivery shape**: When is a custom view sufficient (layout inside the dashboard) and when does it take a full-page custom panel (its own sidebar page)? `ha/lovelace-views-panels` leaves a heuristic open; the skill runs the delivery-shape check case-by-case.
- **`route` property depth**: The `panel` API table lists `hass`, `narrow`, and `panel`, but the example additionally renders `route`. Is `route` guaranteed to be set? Currently the skill accepts it as a SHOULD.
- **`js_url` vs. `module_url`**: Registration supports both classic scripts (`js_url`) and ES modules (`module_url`). The skill requires `module_url` throughout; when `js_url` is still sensible remains open.
- **`embed_iframe` trade-offs**: When is iframe embedding worth it (isolation vs. direct `hass` access)? The skill sets it only on explicit request.
